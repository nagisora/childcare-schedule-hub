# コーディング規約と運用方針

本プロジェクト（Childcare Schedule Hub）における、開発時の基本ルールと運用方針です。AIによる開発支援ツール（Cursor等）を活用した効率的な開発を目指します。

## ディレクトリ構成の原則

- `apps/web/`
  - `app/`: Next.js の App Router 関連（Pages, Layouts, API Routes）
  - `components/`: UIコンポーネント。ロジック（Hooks等）を直接持たず、Propsでの制御（Presentational Components）を基本とする。
  - `hooks/`: React の Custom Hooks。ロジックや状態管理、副作用（localStorage/Supabase等）はここに集約する。
  - `lib/`: 外部API（Supabase, Instagram 等）との通信処理、汎用的なユーティリティ関数（Date系, 正規化処理）など。純粋な TypeScript モジュール。
  - `__tests__/`: ユニットテスト（Vitest）。可能な限り対象ファイルと同じディレクトリか、あるいは用途ごとにまとめる。
  - `tests/`: E2Eテスト（Playwright）。
- `apps/scripts/`: バッチや1回きりのスクリプト群。
- `supabase/migrations/`: DB スキーマ定義。ここを正（Source of Truth）とする。

## 開発・コーディングルール

### 1. 「正本」はコードとDB
システム構成や設計の正解は「現在のソースコード」および「Supabase のマイグレーションファイル (`supabase/migrations/*.sql`)」にあります。
AIエージェントに指示を出す際や新しい機能を実装する際は、**必ず現在のコードベース・テーブル構成を `Read` / `Grep` / `Glob` して最新の情報を確認**してから進めてください。

### 2. SRP（単一責任の原則）の徹底
特に Custom Hooks や UI コンポーネントが肥大化（400行以上など）した場合、早めにドメインごとにフックや関数を分割（Facadeパターン等）し、見通しとテスト容易性を維持してください。

### 3. コミット・ドキュメントの言語
**コミットメッセージ、プルリクエスト、ドキュメントはすべて「日本語」** で記述してください。

### 4. 実行基盤とパッケージ管理
- ランタイムのバージョン管理には **`mise`** を使用します（ローカル実行時の任意導入）。
- パッケージマネージャーには **`pnpm`** (>= 10.x) を使用します。`npm` や `yarn` は使用しないでください。
- フォーマッターは **Biome** を段階導入中です。実行可能な箇所では `pnpm format:check` および `pnpm format` を活用してください。Linterは現在ESLintを利用しています。

## AI (Cursor) 運用方針

- ドキュメントを「AIへの事前コンテキスト用」に過剰に作成・メンテナンスするのは廃止しました。
- 各作業（バグ修正・新機能追加・リファクタ）を行う前に、エージェント（AI）自身がコードの現状を検索し、文脈を理解する**自己解決型の探索**を推奨します。
- 一度に大規模な変更を行わず、**影響範囲を特定し、小さく分割されたステップ（PR）単位で実装**を進めることを原則とします。
