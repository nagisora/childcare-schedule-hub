# ドキュメント整理と再構築実行計画

## 背景

現在、`docs/` 以下に散乱している `01`〜`20` などのファイル群や、過去の `phase-planning`、`dev-sessions` などの管理ドキュメントが、現在の実際のコード・DBの構成と乖離してきており、AI・人間双方にとって「どれが最新の正解か」を判断する際のノイズとなっています。

## 解決方針

1. **不要ディレクトリの完全削除**
  使われなくなったテンプレート管理ディレクトリ（`dev-sessions`, `phase-planning`）を削除し、運用ルールを断捨離します。
2. **古いドキュメントのアーカイブ化**
  現在 `docs/` 直下に存在するナンバリングファイル群（`01-requirements.md` 〜 `20-deferred-work.md` など）や `phase-artifacts` を、すべて `docs/archive/` ディレクトリに退避します。
3. **最新状態に基づく新ドキュメントの生成**
  コードベースとDBマイグレーションファイルから「現在の真のシステム構成・スキーマ」を抽出し直し、必要最小限の最新ドキュメントとして `docs/architecture/` 等に再構築します。

## 対象ファイルとディレクトリ

### 削除対象

- `docs/dev-sessions/` (ディレクトリごと)
- `docs/phase-planning/` (ディレクトリごと)

### アーカイブ（移動）対象

- `docs/01-requirements.md` 〜 `docs/20-deferred-work.md` (全ナンバリングファイル)
- `docs/phase-artifacts/` (ディレクトリごと)
- `docs/announcements/` (ディレクトリごと)

### 新規作成・更新対象

- `docs/README.md` (ポータルとして刷新)
- `docs/architecture/system-design.md` (全体構成図・技術スタック)
- `docs/architecture/database-schema.md` (最新DB定義)
- `docs/rules/coding-guides.md` (最低限の規約)

## 実施ステップ

### 1. 退避用ディレクトリの作成とアーカイブ移動

`docs/archive/` を作成し、不要な長大ドキュメント・過去の遺物をすべて格納します。

### 2. テンプレート管理ディレクトリの削除

`dev-sessions`、`phase-planning` を完全に削除します。

### 3. 最新スキーマ・アーキテクチャ情報の抽出と執筆

- `supabase/migrations/` 内の SQL ファイルと `apps/web/lib/types.ts` 等を読み込み、現在のテーブル構造（`facilities`, `schedules`, `facility_schedules` 等）とリレーションを抽出します。
- `docs/architecture/database-schema.md` に簡潔なテーブル定義を記述します。
- `docs/architecture/system-design.md` に Next.js (App Router), Supabase, Vercel といった現在のアーキテクチャの全体像を記述します。
- `docs/rules/coding-guides.md` に Cursor/AI を用いた現在の運用方針やディレクトリの基本構成を記述します。

### 4. ポータル (`docs/README.md`) の再構築

古いファイルへのリンクをすべて削除し、新設した `architecture/` や `rules/` のドキュメントへのリンクだけを配置するスリムな README に刷新します。

### 5. プロジェクト内リンクのチェック

リポジトリ全体（`README.md` 等）で古いドキュメント（`docs/01-*.md` 等）へのリンクが残っていないか `Grep` ツール等で確認し、あれば適宜外します。
